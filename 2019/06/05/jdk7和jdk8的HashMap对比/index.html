jdk7和jdk8的HashMap对比

发表于 2019-06-05 | 分类于 java

   在jdk7 和jdk8中，HashMap和ConcurrentHashMap的源码做了很多调整，这里就先比较下HashMap的变化。
   每个容器都会从它的初始化，put，get还有size这几个角度来进行分析。

HashMap

JDK7

关键词：数组+链表

   在1.7的版本中，HashMap的内部结构是一个数组+链表。内部用一个Entry<K,V>的数组来保存数据，数组中保存一个链表。这个Entry主要属性有hash,key,value.下面是他的数据结构：

   HashMap有两个参数会影响其性能：
initialCapacity：容器的默认容量，也就是内部hash桶的数量,默认值为16.
loadFactor：扩容因子，默认值为0.75，
扩容的判断通过容量和扩容因子共同决定

初始化

   在HashMap内部通过数组+链表的形式来保存数据，初始化的过程也就是对这个内部hash桶的一个初始化过程。

此时内部用来保存数据的对象Entry<K,V>还是一个空的数组：

transient Entry<K,V>[] table = (Entry<K,V>[]) EMPTY_TABLE;

static final Entry[] EMPTY_TABLE = {};

HashMap的退化

   通过hash算法，可以使得key尽量均匀地分布在所有的hash桶上，但凡是总有例外，当所有的key通过计算都分布到同一个hash桶上的时候，这个时候虽然我们使用的数据结构是HashMap，但其实它已经退化成了链表，在这种情况上，对容器的性能有很大的影响。

扩容

   上面说的那种极端的情况很少会出现，绝大多数时候在数据达到一定标准以后，HashMap内部就会通过扩容来解决这种key集中分布的情况：

扩容需要同时满足两个要求：

  1. 当前的数据量达到阈值；
  2. 即将要插入的位置已经有数据；

这个判断是在put动作执行之前的，这样可以避免无效扩容，满足扩容条件以后，通过resize操作会进行扩容：

扩容会将原有的容量直接扩大一倍，同时通过transfer会对旧数据重新计算hash值，分配新的空间，所以数据量越大，扩容越消耗性能。这也是为什么在使用HashMap的时候建议直接初始化容量的原因：

HashMap<String, String> map = new HashMap<>(128);

size

   因为HashMap是非线程安全的，所有它的size没有任何处理，就是一个全局变量，只要有数据put成功了，就进行+1。

transient int size;

JDK8

关键词：数组+链表+红黑树

   在1.7版本的HashMap中，当key通过hash计算分布到同一个hash桶的数据量过多时，也就是所说的hash碰撞的节点较多，这个时候的性能会变差，为了优化这种情况，1.8以后的HashMap内部增加了红黑树来处理，当Hash碰撞的节点超过8个时，会将这个链表转化为红黑树来保存。

初始化

   大体上没有变化，细节调整了下，默认容量在初始化时就会被调整，一样选择距离设置容量最近的2次幂。

我用不同颜色框处对于不同场景的处理：

红色：首次put时触发，完成初始化；
绿色：这个hash槽尚未被占用，直接将数据放在这个位置
深蓝色：对于重复key的处理，也就是覆盖原来的值；
淡蓝色：当前槽上的数据已经被转化成了红黑树，调用树的处理增加节点；
黄色：把数组直接插到当前链表的尾部，如果hash碰撞的节点超过8个，将这个链表转为红黑树处理；
白色；数据插入以后，判断数据量是否达到阈值标准，是的话，执行扩容操作。

扩容

   在jdk8中，扩容操作在put操作完成以后进行，在刚看到这边时，我有一个疑问，在jdk7中，扩容是需要进行rehash的操作，每个对象都需要重新计算hash值，然后重新分配空间，在jdk8中如果也是这样的操作，那么在put完成以后又重新计算了一遍的话，不是对性能有比较大的影响么？

扩容有三个处理场景：

红色部分：对于当前没有占用的hash槽，直接计算空间分配；
绿色部分：红黑树的处理；
黄色部分：根据key的hash值和容量做按位与运算，因为是扩容两倍，所以这两个key的比较结果只要看新增的bit位是0还是1,0的话维持原位不动，1的话则将数据移动至原位+oldcap那个位置去。

总结

   jdk7中的HashMap内部结构为数组+链表，默认容量为16,扩容需满足容量达到阈值且即将插入的数组下标已有数据，大小直接扩大一倍，jdk8中的HashMap内部结构为数组+链表+红黑树，当链表上hash碰撞的节点超过8个时，就会将链表转为红黑树，保证数据查询的效率，只要达到达到阈值就会扩容，直接扩大一倍，每次扩容都伴随着rehash的过程，比较耗费性能，所以在知道数据大小的情况下，直接设置初始值是比较好的选择。
